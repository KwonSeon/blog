# P0-015-FE-PUB-1 v0.dev Prompt

아래 요구사항을 만족하는 **블로그 플랫폼 홈 화면 UI**를 만들어줘.

중요:
- 지금 단계는 **UI만 먼저 구현**한다.
- API 연동, 서버 액션, 실제 데이터 fetching, 인증, 상태관리, 업로드, 필터 동작은 넣지 않는다.
- 더미 데이터와 정적 props만 사용한다.
- 이 결과물은 **참고용 UI 코드 산출물**이다.
- 실제 프로젝트 생성, 초기 세팅, 의존성 설치, 실행 환경 구성은 사용자가 직접 진행한다.
- 따라서 응답은 **새 프로젝트를 만드는 안내**보다 **파일 단위 UI 코드 제안**에 집중해야 한다.
- 결과물은 **Next.js App Router + TypeScript + Tailwind CSS** 기준의 코드여야 한다.
- 구조는 **FSD(Feature-Sliced Design)** 원칙을 따르되, 이 단계에서는 `app / widgets / entities / shared` 중심으로만 구성한다.
- 이 프로젝트는 **블로그/공개 페이지이므로 SSR/SEO 친화적인 구조를 전제로** 작성해야 한다.
- 즉, 홈 화면은 **client-only 페이지로 만들지 말고**, 가능한 한 **Server Component 기본**으로 구성해야 한다.
- 지금은 홈 UI만 만들지만, 이후 **다크모드 / 다국어(i18n) / 광고(AdSense) / 신규 기능 / 신규 화면**이 자연스럽게 추가될 수 있는 구조와 디자인 언어를 전제로 해야 한다.
- 이미 존재하는 디자인 시스템에 맞추는 작업이 아니라, 이 프로젝트의 첫 공개 화면을 설계하는 작업이다.
- 흔한 SaaS 대시보드 느낌 말고, **개인 기술 블로그 + 프로젝트 쇼케이스 랜딩** 성격이 분명해야 한다.
- **한국어 UI**로 작성한다.
- **화면 코드만 생성**하고, 백엔드/infra 코드는 만들지 않는다.

## 1. 프로젝트 맥락

이 프로젝트는 개인 블로그/프로젝트 플랫폼이다.

핵심 목표:
- 개발 성장 기록(70%) + C 가이드/튜토리얼(30%)
- 프로젝트 홍보 및 “직접 써보기” 경험 제공
- SEO에 유리한 공개 화면 구성

홈 화면의 핵심 역할:
- 방문자가 들어오면 먼저 **프로젝트**를 둘러볼 수 있어야 한다.
- 동시에 **블로그 글**도 별도 섹션으로 분리해서 보여줘야 한다.
- 즉 홈은 “글 목록 포털”이 아니라 **프로젝트 랜딩 + 블로그 허브** 역할이어야 한다.

이번 단계의 로드맵 이름:
- `P0-015-FE-PUB-1 홈 UI(프로젝트/블로그 섹션 분리)`

즉 이번 결과물은:
- 홈 한 화면에서
  - 프로젝트 섹션
  - 블로그 섹션
  - 상단 소개/가치 제안
를 명확히 분리해 보여주는 UI여야 한다.

## 2. 기술 스택과 구현 기준

이 화면은 아래 스택을 전제로 작성해줘.

- Framework: `Next.js 16 App Router`
- React: `React 19`
- Language: `TypeScript`
- Styling: `Tailwind CSS`
- UI primitives: `shadcn/ui`
- Icons: `lucide-react`
- Utility: `clsx`, `tailwind-merge`, `class-variance-authority`

추가 기준:
- `shadcn/ui`의 primitive를 화면에 직접 흩뿌리는 방식보다, **우리 서비스 톤에 맞는 공통 UI 컴포넌트로 한 번 감싼 구조**를 선호한다.
- 즉, 이후에도 재사용 가능한 `shared/ui` 레이어를 염두에 두고 작성해라.
- 예를 들어 `SectionHeader`, `PageContainer`, `SurfaceCard`, `CTAButton`, `PromoSlot` 같은 공통 UI는 별도 컴포넌트로 분리 가능한 형태가 좋다.
- 단, 모든 것을 과하게 추상화하지는 말고, **실제로 반복 사용될 가능성이 높은 UI만 공통 컴포넌트화**한다.

이번 단계에서는 사용하지 않는다:
- TanStack Query
- Zustand
- React Hook Form
- API fetching
- server actions
- Suspense 기반 데이터 로딩

즉, **정적 더미 데이터 기반의 첫 화면 UI**만 만든다.

추가 기준:
- 홈 화면은 **SSR/SSG에 자연스럽게 올릴 수 있는 구조**로 작성한다.
- 데이터가 없더라도 **crawl-friendly markup**이 나오도록 텍스트와 링크 중심으로 구성한다.
- `metadata`를 붙이기 쉬운 구조를 고려하되, 이번 단계에서는 실제 SEO 로직 구현보다 **SEO-friendly UI 구조**에 집중한다.
- 불필요한 client state, browser-only API, hydration 의존 패턴을 최대한 피한다.
- 실제 실행 가능한 완성 앱을 보장하는 것이 아니라, **현재 repo에 이식 가능한 참고용 UI 코드**를 주는 것이 목적이다.
- 디자인 토큰과 레이아웃 규칙은 **향후 다크모드, 다국어, 광고, 신규 화면 확장**까지 고려한 방향이어야 한다.

## 3. FSD 구조 요구사항

FSD를 지키되 과하게 쪼개지 말고, 홈 화면 기준으로만 적절하게 나눠줘.

원하는 구조 예시:

```text
src/
  app/
    (public)/
      page.tsx
  widgets/
    home-hero/
      ui/home-hero.tsx
    home-projects/
      ui/home-projects-section.tsx
    home-posts/
      ui/home-posts-section.tsx
  entities/
    project/
      model/types.ts
      ui/project-card.tsx
    post/
      model/types.ts
      ui/post-card.tsx
  shared/
    config/site.ts
    lib/mock/home-data.ts
    ui/section-header.tsx
    ui/container.tsx
```

원칙:
- `app`은 조립만 한다.
- `widgets`는 홈 섹션 단위 UI를 가진다.
- `entities`는 `project`, `post` 카드 단위 표현을 가진다.
- `shared`는 범용 UI/설정/더미 데이터를 둔다.
- `features`는 이번 단계에서는 굳이 만들지 않는다.
- `shared/ui`에는 shadcn primitive를 서비스 톤으로 감싼 **재사용 가능한 공통 UI 컴포넌트**를 둘 수 있어야 한다.
- **폴더 구조도 반드시 FSD 스타일로 제안/출력**해야 한다.
- 즉, 한 파일에 모든 UI를 몰아넣지 말고 `app -> widgets -> entities -> shared` 계층이 드러나야 한다.
- `page.tsx`는 데이터를 직접 정의하지 말고, `shared/lib/mock/home-data.ts`와 `widgets`를 조합하는 역할만 가진다.
- 실제 경로 루트가 `src/`인지 아니면 프로젝트에서 조정될지는 중요하지 않다.
- 중요한 것은 **FSD 계층이 드러나는 파일 분리 방식**과 **각 파일의 역할**이다.
- 지금은 `features`를 최소화하더라도, 이후 `features / pages / processes` 계층이 늘어나도 무리 없이 확장될 수 있는 naming과 책임 분리를 고려한다.

## 3-1. SSR/SEO 구조 요구사항

이 화면은 블로그 공개 홈이므로 SEO를 고려한 구조여야 한다.

반드시 반영할 것:
- `page.tsx`는 기본적으로 **Server Component**
- 카드/섹션도 굳이 필요하지 않으면 `use client`를 붙이지 않음
- 실제 데이터 연동이 들어와도 `SSR/SSG`로 자연스럽게 확장 가능한 구조
- 텍스트 기반 정보 구조가 먼저 보이도록 설계
- 프로젝트/글 섹션 모두 **실제 링크 요소**를 포함
- heading hierarchy를 명확히 유지
- 추후 `generateMetadata`나 `metadata export`를 붙이기 쉬운 페이지 구조

지양할 것:
- 전체 페이지를 하나의 `use client` 컴포넌트로 만드는 방식
- 시각 효과 때문에 핵심 텍스트가 늦게 렌더링되는 구조
- 링크보다 버튼 중심으로만 탐색을 구성하는 방식
- SEO를 해치는 과도한 인터랙션 중심 랜딩

## 4. 디자인 방향

디자인 톤은 아래를 따른다.

- 기술 블로그지만 너무 딱딱한 문서형이 아니라, **프로젝트를 먼저 경험하게 하는 랜딩 페이지** 느낌
- 동시에 블로그 섹션은 “기록/아카이브” 감성이 보여야 함
- 전체 분위기는 **정제된 에디토리얼 + 제품 쇼케이스**의 중간 지점

피해야 할 것:
- 보라색 중심의 흔한 AI SaaS 톤
- 흰 바탕에 회색 박스만 놓인 템플릿 느낌
- 관리자 대시보드 같은 밀도
- 카드만 잔뜩 늘어놓은 커머스 느낌

원하는 시각 방향:
- 따뜻한 중성 계열 배경
- 짙은 녹색/먹색 계열 포인트
- 종이 질감이 느껴지는 부드러운 배경 또는 그에 준하는 그라데이션/레이어
- 타이포그래피 중심 레이아웃
- 프로젝트 섹션은 실용적이고 선명하게
- 블로그 섹션은 읽고 싶게 만드는 텍스트 위주 톤

예상 컬러 방향 예시:
- background: warm ivory / paper
- foreground: deep charcoal
- accent: dark green / muted gold
- border: soft neutral line

기본 시안은 **라이트 테마**다.

하지만 반드시 고려할 것:
- 지금 당장 다크모드 토글을 구현하지는 않더라도, **추후 다크모드가 붙어도 무너지지 않는 색상 구조**를 전제로 한다.
- 색상은 가능하면 토큰/변수 중심으로 생각하고, 하드코딩된 단일 조합에 과도하게 의존하지 않는다.
- 라이트 테마 기준의 현재 시안이면서도, 다크 테마로 전환했을 때 대비와 계층이 유지될 수 있어야 한다.

## 5. 화면 정보 구조

홈 화면은 최소 아래 순서를 가져야 한다.

1. 상단 Hero
2. 대표 프로젝트 섹션
3. 최신 글 섹션
4. 하단 CTA 또는 짧은 소개 블록

추가 고려사항:
- 중간 또는 하단에 **향후 Google AdSense 슬롯이 삽입될 수 있는 자리**를 UX상 어색하지 않게 둘 수 있어야 한다.
- 지금 단계에서는 실제 광고 코드를 넣지 않고, **광고/프로모션 블록이 들어가도 무너지지 않는 레이아웃 구조**만 고려하면 된다.

### 5-1. Hero 섹션

역할:
- 이 사이트가 “무엇을 하는 곳인지” 한 번에 설명
- 프로젝트와 블로그의 관계를 명확히 보여줌

Hero에 꼭 포함할 것:
- 짧고 강한 헤드라인
- 한두 문장의 설명
- CTA 2개
  - `프로젝트 보기`
  - `글 보러가기`
- 운영자의 포지셔닝이 드러나는 짧은 문구
  - 예: “개발 기록과 직접 만든 서비스를 함께 공개합니다”

Hero는 너무 거대한 마케팅 사이트처럼 과장하지 말고, 개인 플랫폼 특유의 신뢰감이 느껴져야 한다.

추가 가능 요소:
- 작은 상태 배지
- 기술 키워드
- “최근 집중 분야” 짧은 라벨

### 5-2. 프로젝트 섹션

이 섹션이 홈의 첫 번째 메인 콘텐츠다.

역할:
- 방문자가 가장 먼저 프로젝트를 둘러보게 함
- 각 프로젝트는 “직접 써보기” 느낌을 줘야 함

프로젝트 카드에 포함할 것:
- 커버 영역(이미지 자리, 현재는 placeholder 가능)
- 프로젝트 이름
- 한 줄 설명
- 태그 2~4개
- 상태 배지 예시
  - `운영 중`
  - `실험 중`
  - `아카이브`
- CTA
  - `자세히 보기`
  - `서비스 보기` 또는 `데모 보기`

프로젝트 섹션 상단에는:
- 제목
- 짧은 소개 문구
- `전체 프로젝트 보기` 링크

레이아웃:
- 데스크톱에서는 2~3열 카드
- 모바일에서는 1열
- 첫 카드 하나 정도는 강조 스타일을 써도 좋다
- 추후 프로젝트 목록/상세 화면으로 확장될 때도 같은 카드 언어를 재사용할 수 있어야 한다

### 5-3. 블로그 섹션

역할:
- 프로젝트 중심 홈 안에서도 글 아카이브가 분명히 보이게 함
- 단순 목록이 아니라 “읽을 이유”가 보이게 해야 함

글 카드에 포함할 것:
- 카테고리 또는 태그 라벨
- 제목
- 2줄 내외 요약
- 발행일
- 읽기 시간 예시 텍스트
- 관련 프로젝트가 있으면 작은 연결 라벨

블로그 섹션 상단에는:
- 제목
- 짧은 소개 문구
- `전체 글 보기` 링크

디자인 톤:
- 프로젝트 섹션보다 텍스트 비중이 높아야 함
- 아카이브/에세이/튜토리얼 느낌이 살아야 함
- 추후 글 목록/상세, 검색/필터 화면으로 확장될 때도 같은 정보 구조를 재사용할 수 있어야 한다

### 5-4. 하단 CTA 또는 소개 블록

역할:
- 방문자에게 다음 행동을 제안

예시:
- “프로젝트를 먼저 둘러보거나, 글로 정리된 배경을 읽어보세요”
- “C, 시스템, 웹 개발 기록을 꾸준히 정리합니다”

이 영역은 과한 뉴스레터 폼 대신, 단정한 마무리 블록이면 충분하다.

또한 아래 요소가 후속으로 붙어도 레이아웃이 무너지지 않아야 한다:
- 광고 블록
- 추천 콘텐츠
- 프로필 소개
- 구독 CTA
- 다국어 전환 UI

## 6. 더미 데이터 요구사항

실제 API 연동은 하지 말고, `shared/lib/mock/home-data.ts`에 더미 데이터를 정의해 사용해줘.

더미 데이터는 아래 성격을 반영해줘.

프로젝트 예시 톤:
- 주식 대시보드
- 블로그/미디어 플랫폼
- 실험적 사이드 프로젝트

글 예시 톤:
- C 튜토리얼
- 시스템 프로그래밍 메모
- Spring/Next.js 설계 기록
- 프로젝트 회고

프로젝트는 3~4개, 글은 4~6개 정도면 충분하다.

## 8. 접근성과 반응형 요구사항

## 9. 코드 스타일 요구사항

- TypeScript strict-friendly 코드
- props 타입 명확히 정의
- 컴포넌트 이름은 역할 중심
- 불필요한 `use client` 최소화
- 가능하면 server component 기본 유지
- mock data 기반이므로 상태 훅 최소화
- `metadata`, `semantic layout`, `링크 구조`를 붙이기 쉬운 공개 페이지 스타일 유지
- 텍스트는 향후 i18n 리소스로 분리하기 쉬운 방식으로 다룬다
- 헤더, 버튼, 카드 폭을 과도하게 고정하지 않아 다국어 길이 변화에 버틸 수 있어야 한다

shadcn/ui는 필요한 최소 컴포넌트만 사용해도 된다.
오히려 전부 shadcn 카드로만 덮지 말고, **레이아웃과 타이포그래피 자체로 개성을 만들 것**.

## 10. 최종 산출물 요구사항

아래를 만들어줘.

1. 홈 페이지 엔트리 파일
2. Hero 위젯
3. 프로젝트 섹션 위젯
4. 블로그 섹션 위젯
5. 프로젝트 카드 entity UI
6. 글 카드 entity UI
7. mock data 파일
8. 공용 레이아웃 보조 컴포넌트 1~2개
9. 광고 또는 프로모션 슬롯이 들어갈 수 있는 UI 자리 제안

추가로 원한다:
- 코드가 너무 장황하지 않게
- 하지만 실제로 바로 프로젝트에 넣어도 될 정도로 구체적일 것
- 화면이 “대충 나온 시안”이 아니라, **첫 공개 홈 화면 시안으로 바로 쓸 수 있는 수준**이어야 한다
- `npm create`, `pnpm create`, `npx shadcn` 같은 프로젝트 생성/설치 명령은 중심이 아니다
- 응답의 핵심은 **UI 코드와 파일 역할 제안**이어야 한다
- 이번 홈 화면은 이후 아래 단계로 확장 가능한 UX/UI 언어를 가져야 한다
  - 프로젝트 목록/상세
  - 글 목록/상세
  - 관리자 화면
  - 이미지 업로드 화면
  - Google AdSense 삽입
  - 다국어 지원
  - 다크모드
- 또한 shadcn/ui 기반이더라도, **우리 서비스 디자인 시스템으로 재사용할 수 있는 공통 UI 레이어를 먼저 제안**해야 한다
- 예: `shared/ui/container`, `shared/ui/section-header`, `shared/ui/surface-card`, `shared/ui/promo-slot`, `shared/ui/cta-button`
- 도메인 카드(`ProjectCard`, `PostCard`)와 공통 표현 컴포넌트를 구분해서 제안해라
